# GATEWAY ECOSYSTEM TEST GUIDE
*The Ultimate A-to-Z Standalone Runbook*

This guide details the complete, step-by-step process to test the entire 3-part ecosystem: the Standalone ESP32 Gateway, the Django Backend, and the Flutter Mobile App on a physical phone. 

Follow these steps exactly, assuming a completely powered-off state at the beginning.

---

## Phase 1: Backend & Cloud Preparation

### 1. Verify Cloud MQTT Credentials
Ensure that your Cloud MQTT broker credentials (Host, Port, Username, Password) match exactly in both:
- **Django Backend:** Usually found in your `settings.py` or `.env` file.
- **ESP32 Code:** Found in your `Config.h` or `main.cpp`.

### 2. Start the Django Server & MQTT Listener
Open two separate terminal windows for your Django project.

**Terminal 1: Start the Django Server (Daphne/Channels)**
Activate your virtual environment, then run:
```bash
python manage.py runserver
# Or if using Daphne directly for WebSockets:
daphne -b 0.0.0.0 -p 8000 your_project_name.asgi:application
```

**Terminal 2: Start the Background MQTT Listener**
In your second terminal (with the virtual environment activated), start your custom management command:
```bash
python manage.py mqtt_listener
```
*(Note: Replace `mqtt_listener` with the actual name of your management command if it differs).*

### 3. Verify Backend Logs
- **Terminal 1 (Django):** Look for logs confirming ASGI/Channels is running, e.g., `Starting ASGI/Daphne version ... on tcp:port=8000`.
- **Terminal 2 (MQTT):** Look for success logs like:
  ```text
  Connecting to MQTT Broker at ...
  Successfully connected to Cloud MQTT broker.
  Subscribed to topic: sensor/+/data
  Listening for incoming MQTT messages...
  ```

## Phase 2: ESP32 Gateway Boot & Flash

### 1. Enable Test Mode
Before flashing the ESP32, open the C++ project in your IDE (e.g., VS Code + PlatformIO). Ensure the following flag is set to `true` to mock the Sensor Node data:
```cpp
bool TEST_MODE = true;
```

### 2. Connect and Flash
1. Connect the ESP32 Gateway to your computer via USB.
2. Select the correct COM port in your IDE.
3. Build and Upload (Flash) the code to the ESP32.

### 3. Verify Serial Monitor Logs
Open the Serial Monitor and set the baud rate to **115200**. Press the EN/RST button on the ESP32 to restart it. You should see logs indicating a successful boot, Wi-Fi connection, and Cloud MQTT connection:
```text
[WIFI] Connecting to SSID...
[WIFI] Connected! IP: 192.168.x.x
[MQTT] Connecting to Cloud MQTT Broker...
[MQTT] Connected successfully!
[MQTT] Subscribed to topic: gateway/+/commands
```

---

## Phase 3: Physical Phone Setup (Flutter)

### 1. Build and Run on a Physical Device
1. Connect your physical smartphone to your computer (enable USB Debugging for Android / Developer Mode for iOS).
2. Select your device as the target in your Flutter IDE (or terminal).
3. Run the application:
   ```bash
   flutter run --release
   ```

### 2. Emphasize Global Connectivity
**Crucial Note:** Your physical phone can be strictly on **4G / Mobile Data**. It does **not** need to be on the same Wi-Fi network as the ESP32. This proves that the Cloud MQTT architecture successfully routes data globally over the internet, not just over a local network!

---

## Phase 4: Live Test 1 (Mobile App -> Django -> ESP32)

This test confirms that commands from the mobile app traverse the cloud and reach the physical bin.

### 1. Execute the Action
1. Open the Flutter app on your phone.
2. Navigate to the QR scanner feature.
3. Scan the Bin's physical QR code to trigger the Open command.

### 2. Expected Django Terminal Log
In **Terminal 1 (Django)** or **Terminal 2 (MQTT)** depending on your implementation, expect to see the API request received and the MQTT publish event logged:
```text
[POST] /api/bin/open/ 200 OK
Publishing 'OPEN_BIN' to topic 'gateway/bin_001/commands'...
Successfully published message.
```

### 3. Expected ESP32 Serial Monitor Log
In your ESP32 Serial Monitor (115200 baud), expect the gateway to receive the command and actuate the motor:
```text
[MQTT] Message arrived on topic: gateway/bin_001/commands
[MQTT] Payload: OPEN_BIN
start
```
*(The word `start` indicates the Motor Controller logic has been triggered).*

---

## Phase 5: Live Test 2 (ESP32 -> Django -> Mobile App UI)

This test confirms that telemetry data generated by the bin travels through the cloud to automatically update the mobile app.

### 1. Mock Sensor Payload via `TEST_MODE`
Because `bool TEST_MODE = true;` was set in Phase 2, the ESP32 will automatically publish a mock sensor payload (e.g., `50%` capacity) every 10 seconds.

### 2. Expected Django Terminal Log
In **Terminal 2 (MQTT Listener)**, expect logs showing the payload was received and saved to the database:
```text
[MQTT] Received message on topic 'sensor/bin_001/data': {"capacity": 50}
[DB] Bin capacity updated to 50% for bin_001
[WebSocket] Broadcasting UI update for bin_001 to connected clients...
```

### 3. Visual UI Confirmation on the Phone
Look at the dashboard on your physical phone. 
- You should visually see the UI dynamically updating the bin's capacity to **50%**.
- **Crucial:** This must happen automatically via WebSockets without you manually refreshing or pulling down the screen.

---
**End of Test Runbook.** If all phases pass as described above, your 3-part Cloud MQTT ecosystem is functioning perfectly!
